# GitHub CI/CD Design

## Goal

Add a GitHub Actions workflow that runs code quality checks on every push/PR and creates GitHub Releases with a pre-built binary on version tags.

## File Structure

```
.github/
  workflows/
    ci.yml    # Single workflow file with two jobs: check + release
```

## Job 1: `check`

**Triggers:** `push` (all branches), `pull_request` (all branches).

**Runner:** `ubuntu-latest`

**Steps:**

1. `actions/checkout@v4`
2. `DeterminateSystems/nix-installer-action@main` ‚Äî installs Nix with flakes enabled
3. `DeterminateSystems/magic-nix-cache-action@main` ‚Äî transparent GitHub Actions cache for the Nix store
4. `nix develop --command cargo fmt --check` ‚Äî formatting check
5. `nix develop --command cargo clippy --all-targets -- -D warnings` ‚Äî linting, zero warnings
6. `nix develop --command cargo test` ‚Äî 22 unit tests + 10 doc-tests
7. `nix build` ‚Äî verifies the Nix package derivation builds

All Rust commands run inside `nix develop` to get the full toolchain and system dependencies (alsa-lib, pkg-config, libclang) from the flake's devShell. Step 7 builds the package derivation independently (tests the Nix packaging path).

## Job 2: `release`

**Triggers:** `push` with tags matching `v*` (e.g., `v0.1.0`, `v1.2.3`).

**Dependencies:** `needs: check` ‚Äî the check job must pass before the release job runs. This avoids duplicating the check steps.

**Runner:** `ubuntu-latest`

**Permissions:** `contents: write` ‚Äî required to create GitHub Releases and upload assets.

**Steps:**

1. `actions/checkout@v4`
2. Install system dependencies: `libasound2-dev`, `pkg-config`, `libclang-dev` (Ubuntu packages for building)
3. `dtolnay/rust-toolchain@stable` ‚Äî install the stable Rust toolchain
4. `cargo build --release` ‚Äî build the release binary linked against Ubuntu's system libs
5. Copy `target/release/rsonance` to `rsonance-x86_64-linux`
6. `softprops/action-gh-release@v2` ‚Äî creates a GitHub Release from the tag with `rsonance-x86_64-linux` attached

The release binary is built with `cargo build --release` on Ubuntu (not via Nix) so it links against the system's glibc and alsa-lib, making it portable to most Linux distros. The Nix build is only used in the check job to verify the derivation works. Users of the release binary need `pactl` and `mkfifo` on their system PATH for the receiver to function.

## Caching

`DeterminateSystems/magic-nix-cache-action` automatically caches Nix store paths using GitHub Actions cache. It hooks into Nix's build process and caches built derivations transparently. No manual configuration needed.

First run will be slow (~10-15 min) as it downloads nixpkgs and builds dependencies. Subsequent runs with cache should be ~3-5 min.

## Release Flow

1. Developer tags a commit: `git tag v0.1.0 && git push --tags`
2. The `check` job runs (fmt, clippy, test, nix build)
3. If checks pass, the `release` job runs
4. A GitHub Release is created with tag name as the release title
5. The `rsonance-x86_64-linux` binary is attached as an asset
6. Release body is auto-generated by GitHub (commit list since last release)

## What This Does NOT Include

- No aarch64-linux builds (would need ARM runners or cross-compilation)
- No Cachix (GitHub Actions cache is sufficient)
- No branch protection rules (configured in GitHub settings, not in workflow files)
- No changelog generation (GitHub auto-generates from commits)
- No container image builds
- No deployment steps
